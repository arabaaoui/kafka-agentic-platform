Guide d'Authentification GKE & GCP

Comprendre le passage des clés JSON au mode "Zero-Key" (Workload Identity & Impersonation)

1. Les Concepts Fondamentaux

🛡️ KSA (Kubernetes Service Account)

C'est un badge local à un cluster spécifique. Il permet d'ouvrir les portes à l'intérieur du cluster (Pods, ConfigMaps, Secrets). Il est géré par les RBAC Kubernetes.

🌍 GSA (Google Service Account)

C'est une identité globale (email) reconnue par tout Google Cloud. Elle permet d'accéder aux APIs GCP (Vertex AI, Cloud Storage, GKE API). Elle est gérée par l'IAM Google.

2. Scénario A : Workload Identity (Cas "Orrery")

C'est le mode utilisé par une application qui ne travaille que dans son propre projet. Le lien est direct entre son badge interne (KSA) et son identité Google (GSA).

Donne-moi un token pour GSA 'orrery-bot'." MS->>IAM: Vérifie la confiance (Workload Identity User) IAM-->>MS: Token temporaire (JWT - 1h) MS-->>Pod: Voici ton identité temporaire Pod->>IAM: Utilise le token pour appeler Vertex AI
Note technique : Dans ce cas, le GSA est lié au KSA. C'est l'identité native de l'application.

3. Scénario B : Impersonation (Notre Plateforme)

Notre plateforme est un Orchestrateur. Elle tourne dans un projet "Lab" mais doit agir dans des projets distants (Dev, Prod). On utilise l'impersonnalisation (usurpation d'identité sécurisée).

Configuration IAM requise

  1. GSA Maître : Créé dans le projet Lab (kafka-agentic-platform@phenix-lab...).
  2. GSA d'Environnement : Créé dans chaque projet cible (ex: phenix-dev-reader@phenix-dev...).
  3. Relation de Confiance : Dans le projet cible, accorder au GSA Maître le rôle roles/iam.serviceAccountTokenCreator sur le GSA d'Environnement.
# Commande d'administration (GCP)
gcloud iam service-accounts add-iam-policy-binding \
    GSA_ENV@PROJECT_CIBLE.iam.gserviceaccount.com \
    --member="serviceAccount:GSA_MAITRE@PROJET_LAB.iam.gserviceaccount.com" \
    --role="roles/iam.serviceAccountTokenCreator"

4. Droits RBAC (ClusterRole)

Le ClusterRole specifie ce que le compte a le droit de faire à l'intérieur de Kubernetes. On le lie à l'email du GSA pour que GKE fasse le lien. 

# Exemple de ClusterRoleBinding en Prod
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: phenix-platform-read-binding
subjects:
- kind: User
  name: expert-prod@enterprise-prod.iam.gserviceaccount.com  # <-- L'email du GSA Env
roleRef:
  kind: ClusterRole
  name: view # Rôle lecture seule
  apiGroup: rbac.authorization.k8s.io

5. Résumé de l'Isolation Mission/Env

  • 1

    L'Environnement (Menu Infra)

    On y stocke l'email du GSA cible et le Kubeconfig configuré pour l'usurpation.

  • 2

    La Mission (Orchestrateur)

    Le worker verrouille les agents sur l'environnement de la mission. Impossible d'appeler l'IP de la Prod dans une mission de Dev.

  • 3

    L'Exécution (BaseAgent & Orchestrateur)

    Le backend génère un token "juste-à-temps" via l'identité source (ou impersonation), l'injecte dans un Kubeconfig éphémère dédié à la mission, supprimant ainsi la dépendance aux plugins d'authentification interactifs (gke-gcloud-auth-plugin).

6. Le Patching Dynamique (Just-in-Time Token)

Pour garantir une connectivité fiable dans un environnement conteneurisé (Docker/GKE) sans intervention humaine, la plateforme utilise une technique de "Token Injection".

Pourquoi ce mécanisme ?

Par défaut, un Kubeconfig GKE utilise un plugin exec qui appelle gcloud. Dans un conteneur, cela échoue souvent à cause des permissions sur les dossiers de cache ou de l'absence de session active. Le patching remplace cette méthode par un Bearer Token statique à courte durée de vie.

graph TD Start[Déclenchement Mission] --> GetID[Récupération Identité Source] GetID --> Impersonate{Impersonnalisation ?} Impersonate -- Oui --> TargetToken[GSA Env Token] Impersonate -- Non --> BaseToken[GSA Platform Token] TargetToken --> Patch[Injection dans Kubeconfig] BaseToken --> Patch Patch --> Execute[kubectl / Expert Agent] style Patch fill:#1e3a8a,stroke:#3b82f6 style Execute fill:#064e3b,stroke:#10b981

Kubeconfig Original (Template)

users:
- name: gke-user
  user:
    exec:
      command: gke-gcloud-auth-plugin
      args: ["--cluster", "dev"]

Kubeconfig Patché (Éphémère)

users:
- name: gke-user
  user:
    token: "ya29.A0ARWdV7..." # Injecté dynamiquement
    # Plugin exec supprimé

7. Architecture de l'Orchestrateur Centralisé

L'Orchestrateur centralise le contrôle sans nécessiter le déploiement du backend dans chaque environnement cible. La connexion repose sur l'Identité (IAM) et non sur le Réseau (VPC).

Avantages majeurs :

  • Zéro VPC Peering : Pas besoin de lier les réseaux entre les projets GCP.
  • Maintenance Réduite : Un seul backend à mettre à jour pour toute l'entreprise.
  • Audit Unifié : Tous les accès sont tracés depuis l'identité centrale de la plateforme.

8. Développement Local & Tests

En local, votre identité personnelle (compte @enterprise.com) remplace le GSA Maître.

Étape 1 : Authentification

Générez vos credentials ADC sur votre machine hôte :
gcloud auth application-default login

Étape 2 : Montage Docker

Le fichier docker-compose.yml doit monter le dossier config de gcloud : 

volumes:
  - ~/.config/gcloud:/root/.config/gcloud-host:ro
environment:
  - GOOGLE_APPLICATION_CREDENTIALS=/root/.config/gcloud-host/application_default_credentials.json
Attention : Votre compte personnel doit également avoir le droit TokenCreator sur les GSAs d'environnement pour que les tests fonctionnent depuis votre PC.
