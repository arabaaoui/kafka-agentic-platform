Vue d'ensemble

La Kafka Agentic Platform est un OS agentique spécialisé : à la différence d'un chatbot ou d'un simple script de monitoring, elle orchestre une équipe d'agents IA spécialisés qui s'exécutent de façon autonome dès qu'un incident est détecté, sans intervention humaine initiale. Chaque agent reçoit un rôle précis, des outils adaptés et un système-prompt métier (SKILL.md), puis raisonne, appelle des outils réels (kubectl, Prometheus, Jira) et produit un rapport structuré.

Ce que la plateforme fait : dès qu'une alerte Alertmanager ou un ticket Jira correspond à une règle de filtrage, elle déclenche automatiquement 3 agents experts en parallèle, consolide leurs résultats, produit un audit.md avec des hypothèses classées par probabilité, et enrichit une base de connaissances vectorielle (RAG) pour les investigations futures.

Concepts fondamentaux

Qu'est-ce qu'un OS agentique ?

Un OS agentique est un système d'exploitation pour agents IA : il fournit les services d'infrastructure que les agents consomment — gestion de sessions, isolation, sécurité, audit, mémoire, routing LLM — exactement comme un OS traditionnel fournit des syscalls aux processus applicatifs. Les agents ne se soucient pas de savoir comment obtenir un kubeconfig ou comment chunker un document pour le RAG : l'OS s'en occupe.

Ce n'est PAS

  • Un chatbot Kafka avec mémoire
  • Un script de monitoring qui envoie des alertes
  • Un wrapper LLM autour de kubectl
  • Un outil one-shot (pas de persistance)

C'est

  • Un pipeline d'agents autonomes déclenché par des événements
  • Un système avec mémoire vectorielle qui apprend de chaque mission
  • Une plateforme multi-tenant avec isolation stricte par environnement
  • Un auditeur traçable : chaque action est loggée en JSONL

Méthodologie Plan-and-Solve

Tous les agents investigateurs (kafka_strimzi_expert, k8s_gcp_sre, prom_alerts_triage) reçoivent dans leur system prompt le bloc _INVESTIGATION_METHOD qui impose une discipline de raisonnement :

Plan-and-Solve

Avant chaque appel d'outil, l'agent doit articuler : "Je vais appeler [outil] pour vérifier [hypothèse spécifique]." Aucun appel d'outil ne peut se faire sans verbaliser l'intention.

Résilience sur erreur

Une erreur d'outil (timeout, kubectl unreachable) est une découverte, pas une raison de s'arrêter. L'agent la documente et continue avec les outils disponibles.

Approfondissement sur anomalie

Chaque anomalie trouvée (pod Pending, lag anormal, PVC en erreur) déclenche au moins un appel d'outil supplémentaire pour comprendre sa cause.

Auto-Challenge

Avant toute conclusion : "Cette preuve peut-elle signifier autre chose ?" Si pas de cross-validation → appel d'outil de vérification obligatoire.

Politique de langue (ADR-005)

PLATFORM_LANG=fr — Les agents raisonnent en anglais en interne (décisions, PromQL, tool calls) mais rédigent leurs rapports finaux (audit.md, BRIEF.md, cartes KB) en français professionnel. Les termes techniques Kafka/K8s restent en anglais : broker, partition, topic, lag, ISR, URP, PVC, pod, namespace, P99, GC pause.

Architecture globale

Le flux de bout en bout : les déclencheurs (Jira, Alertmanager) alimentent le Filter Engine qui décide si un événement mérite une mission. Si oui, le trigger est inséré dans la table triggers (PostgreSQL) qui joue le rôle de file durable — les workers réclament les triggers via SELECT FOR UPDATE SKIP LOCKED et exécutent les agents via Google ADK, puis les sorties sont persistées et indexées dans le RAG pgvector. N workers async parallèles (WORKER_CONCURRENCY) traitent les missions en concurrence et survivent aux redémarrages du process (at-least-once, dédup via UNIQUE(tenant,source,external_id)).

Cycle de vie complet d'une mission

1

Réception du déclencheur

Un ticket Jira (via JiraMcpPoller) ou une alerte Alertmanager (webhook POST ou poller HTTP) arrive dans l'API. Le payload est enregistré dans la table triggers avec son source et external_id.

2

Évaluation FilterEngine

Le FilterEngine évalue le payload contre les règles actives triées par priorité. La première règle correspondante gagne (first-match wins). Toutes les évaluations sont loggées dans filter_match_log.

3

File durable — claim par un worker

Si une règle matche, le trigger est enregistré dans la table triggers avec matched=true et claimed_at=NULL — il entre dans la file durable PostgreSQL. Les N workers (WORKER_CONCURRENCY, défaut 1) tournent en arrière-plan et appellent claim_next() en continu : un SELECT FOR UPDATE SKIP LOCKED garantit qu'un seul worker réclame chaque trigger. Si un worker crash avant mark_processed(), le trigger est automatiquement re-réclamé à l'expiration du lease (LEASE_SECONDS=900).

4

Classification — IntakeAgent

L'IntakeAgent (LLM pur, sans outils) analyse le payload et retourne un dict avec tenant, env, cluster, type, subject. Si ambiguïté sur l'env ou hors périmètre → statut ignored ou env_ambiguous.

5

Création du MissionContext

Un MissionContext immuable est créé avec un Mission ID structuré (TENANT-ENV-TYPE-SUBJECT-YYYYMMDD-SEQ). Une ligne est insérée dans la table missions et le trigger est linké via mission_id.

6

Isolation GKE — token per mission

Pour les envs GKE, un token ADC/impersonation est demandé dynamiquement et injecté dans un kubeconfig matérialisé dans audits/{mission_id}/kubeconfig.yaml. Chaque mission a son propre kubeconfig, aucun token partagé entre missions.

7

Enquête parallèle — 3 agents experts

Les trois agents (KafkaStrimziExpert, K8sGcpSre, PromAlertTriage) sont lancés simultanément via asyncio.gather() sans stagger — exécution vraiment concurrente. Chaque agent possède ses propres plugins ADK isolés (9 BasePlugin via build_plugin_list()), sa propre AsyncSession SQLAlchemy (socket DB indépendant), et son propre log JSONL. Voir la section ADK — pourquoi asyncio.gather et pas ParallelAgent.

8

Consolidation des preuves

L'EvidenceConsolidatorAgent lit les fichiers *.md produits par les experts (depuis agent-outputs/{mission_id}/) et produit un audit.md avec des hypothèses classées par probabilité et niveau de confiance.

9

Post-Mortem & RAG (auto-déclenché)

Dès que le statut de mission est CLOSED ou PARTIAL, le PostMortemAgent se déclenche automatiquement : il génère un BRIEF.md (5 sections), crée ou met à jour une carte KB dans kb/incidents/, et ingère le tout dans pgvector via ingest_audit() et ingest_kb_card().

Cycle de vie d'un trigger (file durable)

# 1. Insertion par le producteur (idempotente)
INSERT INTO triggers ... ON CONFLICT (tenant, source, external_id) DO NOTHING

# 2. Claim atomique par un worker
WITH next AS (
  SELECT id FROM triggers
  WHERE matched=true AND processed_at IS NULL
    AND (claimed_at IS NULL OR claimed_at < now() - make_interval(secs=>900))
    AND (last_error IS NULL OR last_error NOT LIKE 'DEAD:%')
    AND attempts < 3
  ORDER BY received_at FOR UPDATE SKIP LOCKED LIMIT 1
)
UPDATE triggers SET claimed_at=now(), claimed_by='worker-123-0', attempts=attempts+1
WHERE id=(SELECT id FROM next) RETURNING *;

# 3a. Succès → mark_processed()
UPDATE triggers SET processed_at=now() WHERE id=:id

# 3b. Échec → mark_failed() → reset pour retry
UPDATE triggers SET claimed_at=NULL, claimed_by=NULL, last_error='...' WHERE id=:id

# 3c. Exhaust → mark_dead()
UPDATE triggers SET last_error='DEAD:...', processed_at=now() WHERE id=:id

Filter Engine

Le core/filter_engine.py est le gardien de la plateforme : il décide quels événements méritent une mission agentique. Il opère selon le principe first-match wins : les règles sont triées par priorité croissante et la première qui matche gagne. Toutes les évaluations (match ou non-match) sont loggées dans filter_match_log.

Structure d'une FilterRule

@dataclass(frozen=True)

    tenant: str
    scope: str              # "jira" | "alertmanager" | "care"

Log de toutes les évaluations : que la règle matche ou non, une ligne est insérée dans filter_match_log avec (rule_id, matched, reason, payload_hash). Cela permet d'auditer pourquoi un événement a été ignoré ou traité.

Pipeline ADK — Orchestration des agents

Le pipeline est implémenté dans agents/pipeline/orchestrator.py. Il est consommé par agents/pipeline/worker.py — N tâches asyncio en arrière-plan (WORKER_CONCURRENCY, défaut 1) qui pollent la table triggers via claim_next() (SELECT FOR UPDATE SKIP LOCKED) au lieu d'une asyncio.Queue en mémoire. Chaque worker gère son propre backoff adaptatif (2s → 30s) et un heartbeat toutes les 5 min pour les missions longues.

PipelineOrchestrator.handle() — les 9 étapes

Chaque trigger (Jira ou Alertmanager) déclenche un appel à PipelineOrchestrator.handle(). La méthode exécute 9 étapes séquentielles — seule l'étape 7 (les 3 experts) est parallèle. Toutes les étapes partagent le mission_id comme clé d'isolation.

①

IntakeAgent.classify() — LLM pur, session temporaire

L'intake agent reçoit le payload brut (labels Jira ou Alertmanager) et classe le trigger. Il peut retourner {"status": "ignored"} si hors périmètre, ou {"status": "env_ambiguous"} si l'environnement est indéterminable. Dans ce cas, le pipeline s'arrête immédiatement.

Si le cluster n'est pas identifié par le LLM mais que l'env a un seul cluster configuré, l'orchestrateur l'infère automatiquement depuis TenantRegistry.

②

MissionContext.create() + linkage trigger

Crée la ligne missions en base et génère le mission_id (format TENANT-ENV-TYPE-SUBJECT-YYYYMMDD-SEQ). Met ensuite à jour la table triggers : mission_id = mission_id, processed_at = now() pour que l'UI puisse lier le trigger à la mission.

③

EnvConfig résolution — DB first, YAML fallback

L'orchestrateur cherche d'abord dans infrastructure_envs (source de vérité pour les envs dynamiques). Si absent, il fallback sur le registry YAML (tenants/enterprise.yaml). Crée une EnvConfig isolée pour cette mission (deep copy) afin que les modifications runtime (kubeconfig path, token) n'affectent pas les autres missions concurrentes.

④

GKE Token patching (conditionnel)

Si l'env a un kubeconfig_content (cluster GKE), l'orchestrateur génère un token frais : tente l'impersonation du target_gsa_email (Service Account GCP), puis fallback sur ADC local. Le token est injecté dans le YAML kubeconfig en mémoire (user.exec → user.token), et le kubeconfig est matérialisé dans audits/{mission_id}/kubeconfig.yaml.

⑤

build_expert() — isolation complète par agent

Chaque agent reçoit un chemin d'audit unique (audit.{agent_name}.jsonl) et un TenantConfig transient ne contenant que l'env courant — empêchant tout accès cross-tenant. build_plugin_list() enregistre les plugins directement sur le Runner ADK, et le MissionContext transite via session.state["_mission_context"].

⑥

asyncio.gather() — 3 experts en parallèle

Les trois experts (KafkaStrimziExpert, K8sGcpSre, PromAlertTriage) sont lancés simultanément via asyncio.gather() sans stagger artificiel. Chaque agent reçoit sa propre session SQLAlchemy async (non sérialisable — ne transite jamais via session.state). Chaque agent produit un rapport .md dans agent-outputs/{mission_id}/ et persiste ses observations en base.

⑦

EvidenceConsolidator — synthèse sans outils

Lit les 3 rapports experts depuis le filesystem et demande au LLM de produire un audit.md structuré avec les hypothèses classées par probabilité (tableau rankée), les preuves convergentes et un plan d'action. Utilise le mode thinking pour ce raisonnement de synthèse. Met à jour le statut de la mission (CLOSED ou PARTIAL).

⑧

Refetch status + auto-trigger PostMortem

Le MissionContext est immuable (frozen Pydantic). Pour lire le statut final posé par le consolidator, l'orchestrateur re-fetche la mission depuis la base. Si status ∈ {CLOSED, PARTIAL}, le PostMortemAgent est déclenché automatiquement.

⑨

PostMortemAgent — capitalisation + RAG indexing

Produit le BRIEF.md (5 sections), crée ou met à jour la carte KB dans kb/incidents/, régénère kb/INDEX.md, et ingère audit + carte KB dans pgvector via ingest_audit() et ingest_kb_card(). Non-fatal : une exception ici ne bloque pas la mission — le log indique le skip RAG.

Google ADK 2.x — Framework & Concepts

La plateforme est construite sur Google Agent Development Kit (ADK) 2.x, un framework Python open-source conçu pour créer des agents IA composables et production-ready. Ce chapitre couvre : (1) les 5 primitives fondamentales ADK, (2) le cycle de vie end-to-end des composants ADK dans la plateforme (qui crée quoi, à quelle fréquence, jusqu'à quand), (3) le détail de chaque primitive avec exemples de code, (4) le pattern multi-agent : pourquoi asyncio.gather() plutôt que ParallelAgent — choix argumenté avec les concepts techniques impliqués (AsyncSession, new_message, isolation plugins), (5) le récapitulatif d'usage.

Cycle de vie ADK end-to-end — qui crée quoi, à quelle fréquence ?

Avant de plonger dans le code, il est essentiel de comprendre la granularité à laquelle ADK est instancié. Beaucoup de frameworks créent un pipeline d'agents UNE SEULE fois au démarrage du process et l'utilisent pour tous les traitements. La plateforme fait l'inverse : tout est éphémère par mission. Deux éléments seulement survivent au-delà d'une mission : la queue et le worker. Tout le reste est créé à la demande et jeté en fin de mission.

Pourquoi tout est éphémère (sauf la file durable) ?
Deux éléments seulement survivent au-delà d'une mission : les workers asyncio (permanents dans le process FastAPI) et la table triggers en PostgreSQL (persistante même entre restarts). Tout le reste — PipelineOrchestrator, les 6 Runners, les 9 plugins, les InMemorySessionService, les AsyncSession — est créé à la demande et jeté en fin de mission. Ce pattern garantit un état ADK vierge pour chaque mission : pas de fuite de session.state, pas de plugin "corrompu" par une mission précédente. Le coût de recréation est de quelques millisecondes — négligeable pour des missions qui durent 30 à 90 secondes.

Multi-agent : pourquoi asyncio.gather() et pas ParallelAgent ?

ADK 2.x propose une primitive native ParallelAgent pour exécuter N sous-agents en parallèle dans un seul Runner. La plateforme utilise à la place asyncio.gather() avec des Runners indépendants. Ce n'est pas un oubli — c'est un choix technique délibéré, documenté dans specs/010-adk-2x-migration/spec.md (ligne 171) et research.md (Décision 3). Voici les trois raisons techniques.

Raison 1 — Isolation des sessions DB (AsyncSession SQLAlchemy)

Concept : qu'est-ce qu'une AsyncSession SQLAlchemy ?

Une AsyncSession est composée de deux parties :

1. Un socket réseau vers PostgreSQL — géré par asyncpg, le driver Python async pour Postgres. Ce socket TCP est une connexion exclusive : il ne peut traiter qu'une seule requête en vol à la fois. Asyncpg interdit explicitement d'envoyer deux requêtes simultanées sur le même socket.

2. Une identity map — un dictionnaire en mémoire qui cache les objets ORM déjà chargés pour éviter les requêtes redondantes dans la même transaction.

Règle absolue asyncpg : si deux coroutines font await session.execute(...) sur la même AsyncSession simultanément, asyncpg lève :
InterfaceError: cannot perform operation: another operation is in progress

C'est pourquoi chaque agent expert obtient sa propre AsyncSession — son propre socket TCP, ses propres transactions, ses propres commits indépendants.

Dans l'orchestrateur, chaque expert reçoit une session DB fraîche avant d'être lancé : 

# agents/pipeline/orchestrator.py — isolation DB par expert
async def run_expert(agent):
    async with AsyncSession(db_engine) as agent_session:
        # Chaque expert a SA PROPRE AsyncSession → son propre socket TCP vers Postgres
        # model_copy() clone le MissionContext en remplaçant db_session uniquement
        agent_ctx = mission_ctx.model_copy(
            update={"db_session": agent_session}
        )
        result = await agent.investigate(
            agent_ctx, alert_name, alert_namespace, db_conn=agent_session
        )
    return result

# Les 3 sessions sont ouvertes indépendamment — aucune concurrence sur un même socket
await asyncio.gather(*(run_expert(a) for a in experts))
Pourquoi ParallelAgent casserait ça : ParallelAgent fait tourner ses sous-agents dans un seul Runner avec un seul session.state. Si on y stockait mission_ctx.db_session, les 3 sous-agents partageraient la même AsyncSession — crash assuré dès que deux d'entre eux font une requête DB en même temps. La spec 010 research.md l'explicite (ligne 55) : « Les sessions BD ne sont pas sérialisables dans session.state. Elles doivent être fournies via un callable ou un contexte externe. »

Raison 2 — Chaque agent expert reçoit un prompt différent

Concept : qu'est-ce que new_message dans un Runner ADK ?

Quand on appelle runner.run_async(new_message=Content(...)), le new_message est le message utilisateur initial qui démarre la conversation de l'agent pour cette exécution. Il est distinct de l'instruction (system prompt du LlmAgent, fixe à l'instanciation) :

LlmAgent(instruction="Tu es un expert Kafka...") ← fixe, chargé depuis SKILL.md
runner.run_async(new_message=Content(...)) ← dynamique, différent à chaque appel

Limitation de ParallelAgent : il reçoit UN seul new_message et le transmet identique à tous ses sous-agents — impossible de personnaliser le contexte par sous-agent sans wrapper supplémentaire.

Chaque expert de la plateforme a une méthode investigate() qui construit un prompt différent intégrant les paramètres spécifiques à l'alerte et à l'expert :

Pour faire la même chose avec ParallelAgent, il faudrait soit pré-baker les 3 contextes différents dans l'instruction de chaque LlmAgent (mélangeant prompt métier et données runtime dans SKILL.md), soit créer des wrappers intermédiaires — une complexité qui ne vaut pas le gain.

Raison 3 — Isolation des instances de plugins par agent

Concept : pourquoi l'isolation des instances de plugins compte ?

Chaque appel à BaseAgent.run() crée des instances fraîches de tous les 9 plugins via build_plugin_list(). Ce n'est pas par hasard :

AuditPlugin — accumule les entrées d'audit dans une liste interne (_entries: list[dict]), puis les flush dans audits/{mission_id}/{agent_name}/audit.jsonl. Si 3 agents partagent la même instance, toutes leurs entrées se mélangent dans la même liste — l'attribution par agent disparaît.

ResiliencePlugin — maintient un compteur de retries par tool call. Si partagé entre 3 agents parallèles en asyncio, les compteurs s'entremêlent : l'agent K8s peut déclencher le circuit-breaker de l'agent Kafka par erreur.

Avec ParallelAgent : 1 Runner = 1 liste de plugins partagée. Pour retrouver l'isolation, il faudrait refactoriser les 9 plugins pour lire tool_context.agent_name et partitionner leur état interne. Coût de développement élevé, risque de régression élevé sur 9 classes. 
# agents/base.py — BaseAgent.run() : plugins recréés à CHAQUE appel d'agent

RAG & Auto-Learning — Mémoire vectorielle

La plateforme implémente un système RAG (Retrieval-Augmented Generation) qui permet aux agents de bénéficier des connaissances accumulées lors des missions précédentes. Chaque mission terminée enrichit automatiquement la base de connaissances vectorielle.

RAG — le concept

Les LLMs souffrent de deux limites fondamentales pour le diagnostic d'incidents : une date de coupure (ils ne connaissent pas vos incidents internes) et une tendance à halluciner des solutions plausibles mais fausses faute de contexte métier. Le RAG résout cela en enrichissant le prompt avec des passages de documents réels et pertinents récupérés dynamiquement avant chaque appel LLM.

Sans RAG — LLM pur

  • Connaissance générale Kafka uniquement (training data public)
  • Ne sait pas que PVC-saturation a déjà frappé ce cluster en janvier
  • Peut halluciner des métriques Prometheus ou des commandes kubectl
  • Chaque investigation repart de zéro

Avec RAG — ancré sur votre historique

  • Le prompt inclut les 3 passages les plus similaires de vos incidents réels
  • L'agent sait que ce pattern correspond à la carte KB pvc-saturation-kafkahub
  • Les hypothèses sont validées contre des audits passés vérifiés
  • La base s'enrichit automatiquement à chaque mission terminée

Choix du modèle d'embedding — intfloat/multilingual-e5-small

L'embedding transforme un texte en vecteur numérique dense (384 dimensions). Deux textes sémantiquement similaires ont des vecteurs proches en espace cosinus — c'est ce qui permet de trouver les passages pertinents même si les mots exacts diffèrent.

Multilingue Français + Anglais + 100 langues nativement e5-small (anglais only) — incompatible avec PLATFORM_LANG=fr (ADR-005)

Elle expose une API unifiée (SentenceTransformer(path).encode(texts)) compatible avec tous les modèles du Hub. Le singleton lazy dans core/embeddings.py garantit un seul chargement au démarrage, partagé entre l'ingestion (PostMortem) et la recherche (BaseAgent._fetch_kb_context).

Pourquoi RAG dans un contexte d'incident Kafka ?

Sans RAG

  • Chaque mission repart de zéro
  • Les agents ne savent pas que ce pattern PVC a déjà été rencontré
  • Les solutions connues ne sont pas réutilisées
  • La capitalisation repose sur la mémoire humaine

Avec RAG

  • BaseAgent.run() préfixe le contexte KB/audit au prompt de l'agent avant tout appel d'outil (investigateurs + consolidator)
  • Le PostMortemAgent construit une KB de patterns d'incidents
  • Les hypothèses sont validées contre l'historique des audits
  • La plateforme apprend automatiquement de chaque mission

Architecture RAG — deux sources vectorielles

Stratégie de chunking

Le chunker (core/rag_ingest.py::_chunk_markdown) applique une stratégie en 2 passes :

Passe 1 — Split H2

Le document est découpé selon les titres ## Section. Chaque section devient un chunk si elle ne dépasse pas 512 tokens (≈2048 caractères). Cela préserve la cohérence sémantique des sections.

Passe 2 — Fenêtre glissante

Si une section dépasse 512 tokens, elle est re-découpée avec une fenêtre glissante (512 tokens de taille, 51 tokens d'overlap entre chunks consécutifs) pour éviter de couper un raisonnement en deux sans continuité.

Flux d'ingestion — ingest_kb_card() et ingest_audit()

# core/rag_ingest.py — idempotent (delete-then-insert)
async def ingest_kb_card(slug: str, db: AsyncSession) -> int:
    kb_path = Path(f"kb/incidents/{slug}.md")
    content = kb_path.read_text(encoding="utf-8")

    # 1. Supprime les anciens chunks (idempotent)
    await db.execute(delete(KBChunk).where(KBChunk.kb_slug == slug))

    # 2. Chunke le Markdown
    chunk_tuples = _chunk_markdown(content)  # [(title, text), ...]

    # 3. Génère les embeddings en batch
    embeddings = embedding_service.embed_batch([t for _, t in chunk_tuples])

    # 4. Insère les nouveaux chunks avec embeddings pgvector
    db.add_all([
        KBChunk(kb_slug=slug, chunk_index=i, chunk_text=text,
                embedding=embeddings[i], metadata={"section_title": title})
        for i, (title, text) in enumerate(chunk_tuples)
    ])
    RAG_CHUNKS_INGESTED.labels(type="kb").inc(count)
    return count

Chemin de lecture — Injection sémantique lors des investigations

Le chemin de lecture RAG est déclenché automatiquement au démarrage de chaque mission, dans BaseAgent.run(), avant toute décision d'outil. Pour les agents de _RAG_PREINJECT_AGENTS (investigateurs + consolidator), _fetch_kb_context() encode la query (sujet + type), interroge pgvector par distance cosinus, et préfixe le bloc KB au prompt de l'agent — celui-ci raisonne donc avec le contexte des diagnostics passés dès sa première étape de sélection d'outil.

RAGIndex — recherche cosinus asynchrone (core/mem0_bridge.py)

# RAGIndex.search() — UNION ALL sur kb_chunks + audit_chunks
async def search(self, query: str) -> list[RAGResult]:
    # "query: " prefix requis par les modèles e5 (vs "passage: " à l'ingestion)
    query_embedding = embedding_service.embed_query(query)

    kb_select = sa.select(
        sa.literal("kb").label("source"),
        KBChunk.kb_slug.label("ref"),
        KBChunk.chunk_text,
        KBChunk.embedding.cosine_distance(query_embedding).label("distance"),
    ).order_by("distance").limit(self.limit)

    audit_select = sa.select(
        sa.literal("audit").label("source"),
        AuditChunk.mission_id.label("ref"),
        AuditChunk.chunk_text,
        AuditChunk.embedding.cosine_distance(query_embedding).label("distance"),
    ).order_by("distance").limit(self.limit)

    # Fusionne les deux sources, retrie globalement, limite à top-3
    full_query = sa.union_all(kb_select, audit_select).order_by("distance").limit(self.limit)
    results = await self.db.execute(full_query)
    return [RAGResult(source=r.source, ref=r.ref, chunk_text=r.chunk_text, ...) for r in results.mappings()]

Authentification GKE — Workload Identity & GSA Impersonation

Les clusters Kafka Strimzi tournent sur GKE dans des projets GCP distincts (preprod, prod). Pour que les agents experts puissent y exécuter des commandes kubectl, la plateforme doit obtenir un token OAuth2 GCP valide et l'injecter dynamiquement dans un kubeconfig éphémère par mission. C'est le mécanisme Just-in-Time Token Injection, implémenté dans core/gcp.py et déclenché à l'étape ④ de PipelineOrchestrator.handle().

Concepts fondamentaux — KSA vs GSA

KSA — Kubernetes Service Account

C'est un badge local à un cluster spécifique. Il représente l'identité d'un Pod à l'intérieur du cluster K8s. Il est géré par les RBAC Kubernetes (ClusterRole, ClusterRoleBinding). Seul, il ne peut pas appeler les APIs GCP.

Ex : kafka-platform-sa dans le namespace kafka-ops

GSA — Google Service Account

C'est une identité globale reconnue par tout Google Cloud (email @iam.gserviceaccount.com). Elle permet d'appeler les APIs GCP (GKE API, Vertex AI, Cloud Storage…). Elle est gérée par l'IAM Google et peut être liée à un KSA via Workload Identity.

Ex : platform-sa@phenix-lab.iam.gserviceaccount.com

JSON SA Key files — dépréciés par GCP. GCP recommande de ne plus créer ni utiliser de fichiers JSON de clé de Service Account : credentials long-lived, rotation manuelle, risque de fuite. L'approche correcte pour GKE est Workload Identity + impersonation — aucun fichier JSON, aucun secret à gérer.

Le code core/gcp.py essaie bien un JSON key en premier (PLATFORM_GSA_KEY_PATH) — c'est un fallback dev local uniquement. En GKE le fichier n'existe pas, le code tombe directement sur google.auth.default() → Workload Identity.

Architecture orchestrateur centralisé — multi-cluster via IAM

La plateforme tourne dans un cluster LAB central mais doit agir sur des clusters distants (preprod, prod) appartenant à des projets GCP différents. La connexion repose sur l'identité IAM, pas sur le réseau — aucun VPC Peering requis.

Zéro VPC Peering requis. La plateforme n'a pas besoin d'un lien réseau entre le cluster LAB et les clusters preprod/prod. La connexion repose uniquement sur l'identité IAM — le token OAuth2 obtenu via impersonation est accepté directement par l'API Server GKE du cluster cible. Un seul backend à maintenir pour toute l'entreprise.

Flux complet — Workload Identity → impersonation → token injecté

Just-in-Time Token Injection — pourquoi ce mécanisme ?

Par défaut, un kubeconfig GKE utilise un plugin exec qui invoque gke-gcloud-auth-plugin pour obtenir un token frais. Dans un conteneur, ce plugin échoue souvent : absence de session gcloud active, permissions sur les dossiers de cache, ou simplement parce que gcloud n'est pas installé dans l'image.

La plateforme remplace ce plugin par un Bearer Token statique injecté en mémoire, valide 1h. Chaque mission reçoit son propre fichier kubeconfig matérialisé dans audits/{mission_id}/kubeconfig.yaml — jamais partagé entre missions concurrentes.

Kubeconfig original (template)
users:
- name: gke-user
  user:
    exec:
      command: gke-gcloud-auth-plugin
      args: ["--cluster", "preprod"]
      # echoue en conteneur sans session
Kubeconfig patche (ephemere, par mission)
users:
- name: gke-user
  user:
    token: "ya29.A0ARWdV7..."  # injecte dynamiquement
    # plugin exec supprime
    # valide 1h - aucun plugin requis

Implementation — GCPTokenProvider (core/gcp.py)

# _get_base_credentials() — cascade : dev fallback -> WI prod
async def _get_base_credentials(cls):
    master_key_path = os.getenv("PLATFORM_GSA_KEY_PATH", "/app/kube_conf/platform_master_key.json")

    if os.path.exists(master_key_path):
        # DEV uniquement — ce fichier NE DOIT PAS exister en GKE prod/preprod
        return service_account.Credentials.from_service_account_file(master_key_path, scopes=[...])

    # PROD GKE — google.auth.default() -> Workload Identity via metadata server
    # Aucune config supplementaire quand WI est correctement configure
    creds, _ = google.auth.default(scopes=["https://www.googleapis.com/auth/cloud-platform"])
    return creds

# get_token() — token WI direct ou token impersonne
async def get_token(cls, target_gsa_email=None):
    creds = await cls._get_base_credentials()
    if not creds.valid:
        creds.refresh(Request())

    if not target_gsa_email:
        return creds.token  # token natif GSA plateforme

    # Impersonation : creds WI -> token au nom du Target GSA
    target_creds = impersonated_credentials.Credentials(
        source_credentials=creds,
        target_principal=target_gsa_email,
        target_scopes=["https://www.googleapis.com/auth/cloud-platform"],
        lifetime=3600  # 1h — jamais de long-lived token
    )
    target_creds.refresh(Request())
    return target_creds.token

Glossaire

ADK (Agent Development Kit)
Framework Google pour construire des agents LLM. Fournit LlmAgent, Runner, InMemorySessionService, FunctionTool, et le Plugin System (BasePlugin). La plateforme tourne sur ADK 2.x natif — build_plugin_list() enregistre 9 plugins sur le Runner, AutoTracingPlugin émet les spans OTel automatiquement.
AsyncSession (SQLAlchemy)
Connexion DB asynchrone gérée par asyncpg. Composée d'un socket TCP exclusif vers PostgreSQL + une identity map (cache ORM en mémoire). Règle absolue : un seul await session.execute() en vol à la fois sur ce socket. Partager une AsyncSession entre plusieurs coroutines concurrentes provoque InterfaceError: another operation is in progress. C'est pourquoi chaque agent expert reçoit sa propre AsyncSession via mission_ctx.model_copy(update={"db_session": agent_session}).
ParallelAgent (ADK 2.x)
Primitive ADK qui orchestre N sous-agents en parallèle dans un seul Runner avec un seul session.state partagé. Non utilisée dans la plateforme (cf. section Pourquoi asyncio.gather) : incompatible avec l'isolation des AsyncSession, des prompts custom par expert, et des plugins à état interne.
output_key (ADK 2.x)
Mécanisme natif de ParallelAgent : chaque sous-agent publie son résultat dans session.state[output_key]. Non utilisé ici — la plateforme persiste les outputs des experts sur disque (agent-outputs/{mission_id}/*.md), ce qui permet le replay, le post-mortem et la lecture humaine directe sans dépendre de la session ADK.
audit.md
Rapport final produit par l'EvidenceConsolidator. Contient les hypothèses classées par probabilité, les preuves convergentes et un plan d'action.
audit_chunks
Table pgvector contenant les embeddings des audits de missions passées. Permet la recherche sémantique dans l'historique d'incidents.
AutonomyPlugin
Plugin n°8 de la chaîne. Bloque les actions non autorisées selon le niveau d'autonomie (L2 = lecture seule). Lève AutonomyViolation.
BaseAgent
Classe mère de tous les agents. Gère le chargement SKILL.md, le wrapping des outils, la construction du system prompt et la persistance des sorties.
BRIEF.md
Document de capitalisation en 5 sections produit par PostMortemAnalyst : Résumé exécutif, Cause racine, Impact, Actions prises, Leçons apprises.
EvidenceConsolidator
Agent synthèse (sans outils). Lit les 3 rapports experts et produit un audit.md avec hypothèses rankées. Utilise le mode thinking.
FilterEngine
Composant qui évalue les triggers contre les FilterRules. Principe first-match-wins, tri par priority ASC.
FilterRule
Règle de filtrage stockée en DB. Attributs : scope (jira/alertmanager/care), criteria JSONB, priority, poll_interval_seconds.
IntakeAgent
Premier agent du pipeline. Classifie le trigger (LLM pur, sans outils) pour extraire tenant/env/cluster/type/subject.
kafka-agent-toolkit
Bibliothèque Python stateless (uv workspace). Contient les outils toolkit (prom_query, kubectl, etc.), les schémas Pydantic, le skill loader.
KBChunk / kb_chunks
Table pgvector des embeddings de cartes KB. Alimentée par ingest_kb_card() après création/mise à jour d'une carte.
LiteLLM
Couche de routing LLM multi-provider. Permet de bascule entre Gemini, Claude et d'autres modèles sans modifier le code agent.
MissionContext
Dataclass immuable portant toutes les dimensions d'une mission (tenant, env, cluster, type, subject, mission_id, autonomy_level).
MissionIsolationPlugin
Plugin n°7. Bloque les accès cross-environnement. Chaque mission ne peut cibler que son env de création.
multilingual-e5-small
Modèle sentence-transformers (intfloat/multilingual-e5-small). 384 dimensions. Supporte 100+ langues dont FR/EN. Utilisé pour les embeddings RAG.
pgvector
Extension PostgreSQL pour stocker et requêter des vecteurs d'embeddings. Permet la recherche par similarité cosinus sur kb_chunks et audit_chunks.
Plan-and-Solve
Méthodologie d'investigation injectée dans les agents experts. Impose d'articuler l'hypothèse avant chaque appel d'outil.
build_plugin_list()
Fabrique de plugins ADK 2.x — core/plugins.py. Retourne une liste ordonnée de 9 BasePlugin enregistrés sur Runner(plugins=[...]). Remplace l'ancien PluginChain custom (supprimé en Phase 8, 2026-06-11). Signature : build_plugin_list(mission_context, tenant_config, autonomy_level="L2", audit_path=None).
PostMortemAnalyst
Agent de capitalisation déclenché automatiquement après CLOSED/PARTIAL. Produit BRIEF.md + carte KB + ingère RAG.
SKILL.md
Fichier Markdown par agent (frontmatter YAML + corps). Définit le rôle, les responsabilités et les instructions métier. Hot-reload supporté.
TenantRegistry
Singleton qui charge et maintient les configurations tenant (YAML + surcharges DB). Point d'accès unique pour env configs.
