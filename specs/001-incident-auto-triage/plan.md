# Implementation Plan: Incident Auto-Triage (v0 Core)

**Branch**: `001-incident-auto-triage` | **Date**: 2026-05-10 | **Spec**: [spec.md](./spec.md)

## Summary

Build the v0 read-only incident auto-triage pipeline: Jira poller and Alertmanager webhook detect incidents, route them through a Postgres-backed filter engine, create env-locked missions, dispatch three expert agents in parallel (kafka_strimzi_expert, k8s_gcp_sre, prom_alerts_triage) via Google ADK `ParallelAgent`, synthesize findings with `evidence_consolidator` in thinking mode, and deliver a ranked-hypotheses audit Markdown in the Next.js UI within 5 minutes. `MissionIsolationPlugin` enforces env boundaries at every tool call; "Post to Jira" is explicit opt-in only via `c4-atlassian` MCP.

---

## Technical Context

**Language/Version**: Python 3.11+ (backend), TypeScript 5.3+ (Next.js frontend)
**Primary Dependencies**: Google ADK (`LlmAgent`, `ParallelAgent`), LiteLLM (Gemini primary / Claude fallback), FastAPI + SSE, SQLAlchemy 2.x + Alembic, kafka-agent-toolkit (pip workspace sibling), MCP `c4-atlassian`, Langfuse self-hosted, Promptfoo, Mem0, watchfiles
**Storage**: PostgreSQL 17 (missions, triggers, filter_rules, audits, agent_outputs вЂ” SQLAlchemy models + Alembic migrations), Redis (session cache, SSE fan-out), filesystem (`agent-outputs/{mission_id}/`, `audits/{mission_id}/audit.md`, `audit.jsonl`)
**Testing**: pytest (unit + integration), Playwright (e2e), Promptfoo в‰Ґ80% CI gate, grep CI test (zero-secret in audit.jsonl)
**Target Platform**: Linux server (Docker Compose local dev, Kubernetes/GKE preprod/prod)
**Project Type**: Web service (FastAPI backend + Next.js frontend) + agentic pipeline
**Performance Goals**: Mission created within 60s of Jira poll; full audit delivered within 5 minutes of mission creation; Alertmanager webhook matched within 10 seconds
**Constraints**: Read-only v0 (L2 autonomy вЂ” no mutating tools); zero cross-env tool calls; zero secrets in audit.jsonl; all Enterprise-specific values in `tenants/enterprise.yaml` only
**Scale/Scope**: Single tenant (Enterprise), 2 envs (preprod, prod), ~5 missions/day in dogfooding phase, 15 Promptfoo eval cases

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| # | Principle | Status | Evidence |
|---|-----------|--------|----------|
| I | Read-Only v0 вЂ” L2 autonomy | PASS | No mutating tools registered. All 5 toolkit tools (`prom_query`, `lag_correlation`, `pvc_forecast`, `cluster_health`, `promrule_audit`) are read-only. `AutonomyPlugin` enforces L2; any mutating tool registration triggers a constitution violation. |
| II | MissionIsolationPlugin enforced by plugin (not convention) | PASS | `MissionIsolationPlugin` is in the ordered plugin chain and intercepts every tool call at runtime. Cross-env calls are blocked + logged to `audit.jsonl` + surfaced in UI (FR-004, SC-004, Acceptance Scenario 6). |
| III | Post Jira = explicit opt-in only | PASS | `jira_add_comment` via `c4-atlassian` MCP is invoked only on `POST /v1/missions/{id}/post-to-jira` (UI toggle or explicit API call). Default mission state is "internal only". Automated posting is structurally impossible (FR-008, SC-003). |
| IV | Eval suite CI в‰Ґ80% | PASS | 15 calibrated cases in `evals/cases/` (5 lag, 3 promrule, 4 pvc, 3 consolidation). Promptfoo run is a blocking CI check (SC-005). Each new agent/tool in this spec ships with в‰Ґ1 eval case. |
| V | Zero secret leakage in audit logs | PASS | `AuditPlugin` applies redaction list before writing `audit.jsonl`. CI grep test: `grep -iE 'password\|secret\|token\|api_key\|kubeconfig'` must return 0 (FR-009, SC-007). |
| VI | Skills = SKILL.md filesystem | PASS | Each agent directory contains `SKILL.md` as single source of truth. Hot-reload via `watchfiles` in dev. No skill data in Postgres. |
| VII | No Enterprise hardcode in core/agents/triggers/api/ | PASS | All Enterprise-specific values (cluster names, namespaces, Jira project, poll intervals) live exclusively in `tenants/enterprise.yaml`. `intake_agent` reads tenant config at runtime. Second tenant addable via new yaml file only. |
| VIII | Filter rules = Postgres runtime | PASS | `filter_rules` table in Postgres (JSONB `criteria`). Editable via `/settings/filters` UI without restart. Bootstrap rule inserted by migration `0001_initial.sql`, overridable immediately. No filter hardcoded in source or YAML. |

## Agent Pipeline Design

```
Trigger (Jira poller / Alertmanager webhook)
    в””в”Ђв”Ђ FilterEngine (Postgres filter_rules, priority-ordered)
           в””в”Ђв”Ђ intake_agent (LlmAgent)
                  в”њв”Ђв”Ђ Determines env, cluster, type, subject
                  в”њв”Ђв”Ђ Creates Mission (env-locked)
                  в””в”Ђв”Ђ ParallelAgent
                         в”њв”Ђв”Ђ kafka_strimzi_expert (toolkit: lag_correlation, pvc_forecast, cluster_health)
                         в”њв”Ђв”Ђ k8s_gcp_sre          (toolkit: cluster_health, prom_query)
                         в””в”Ђв”Ђ prom_alerts_triage   (toolkit: promrule_audit, prom_query)
                                в””в”Ђв”Ђ evidence_consolidator (LlmAgent, thinking mode)
                                       в””в”Ђв”Ђ audit.md в†’ Postgres audits table
                                              в””в”Ђв”Ђ POST /v1/missions/{id}/post-to-jira (explicit only)
                                                     в””в”Ђв”Ђ c4-atlassian MCP jira_add_comment
```

### Plugin Chain (all agents, ordered)

```
GuardrailsPlugin в†’ ResiliencePlugin в†’ OTelMetricsPlugin в†’ AuditPlugin
в†’ ActivityPlugin в†’ Mem0MemoryPlugin в†’ MissionIsolationPlugin в†’ AutonomyPlugin в†’ ErrorHandlerPlugin
```

- `MissionIsolationPlugin` blocks any tool call whose endpoint env differs from `mission.env`
- `AuditPlugin` writes every tool call to `audit.jsonl` with redaction applied
- `AutonomyPlugin` enforces L2: rejects any mutating tool at registration time

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Agent framework | Google ADK `ParallelAgent` | Native parallel execution, `LlmAgent` composability, already in stack |
| LLM routing | LiteLLM (Gemini primary, Claude fallback) | Multi-provider resilience; single call site via `ResiliencePlugin` |
| Filter engine | Postgres JSONB `criteria`, priority asc, first-match | Constitution VIII; runtime editable via UI; no YAML hardcode |
| Mission isolation | Plugin (not convention) | Constitution II; structural enforcement, not developer discipline |
| Jira integration | MCP `c4-atlassian` (read: poll + write: comment) | Reuse existing credentials; no new auth setup |
| Post-to-Jira gating | API endpoint + UI toggle (no automatic path) | Constitution III; structural impossibility of silent posting |
| Audit storage | `audit.md` on filesystem, path in Postgres | Markdown audits are human-readable; Postgres stores metadata + post status |
| env determination | intake_agent: Jira custom fields > labels > summary regex | v0 fallback regex acceptable per spec assumptions; ambiguous = abort |
| Secret redaction | AuditPlugin pre-write redaction list | Constitution V; CI grep gate as second line of defense |
| Skills | `SKILL.md` per agent directory, watchfiles hot-reload | Constitution VI; git history = changelog |
